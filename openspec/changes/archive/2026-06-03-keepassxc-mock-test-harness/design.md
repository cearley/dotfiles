## Context

Chezmoi's `keepassxcAttribute` template function calls `keepassxc-cli open <database>` and conducts an interactive session to retrieve secrets. In non-interactive environments (no TTY available for the password prompt), this fails with "could not open a new TTY". Previously there was no way to render templates that contain secret lookups without the live database.

The solution exploits two chezmoi config knobs: `keepassxc.command` (override the binary) and `keepassxc.prompt = false` (skip TTY password prompting). With these set, chezmoi calls our mock binary directly with no credential exchange needed.

## Goals / Non-Goals

**Goals:**
- Allow any chezmoi template to be rendered locally without a live KeePassXC database
- Return deterministic, identifiable mock values by default so missing fixtures never cause silent failures
- Let specific entries be pinned to realistic-looking values (SSH keys, PEM headers) via an optional fixture file
- Zero impact on `chezmoi apply` or any live secret-management flow

**Non-Goals:**
- CI/CD integration — this is a local developer tool
- Validating that secret values are structurally correct (e.g. valid PEM, valid SSH key)
- Mocking keepassxc-cli for anything other than `keepassxcAttribute` calls (e.g. password manager UI)

## Decisions

**D1: Config override via `chezmoi dump-config` + jq rather than PATH injection**

`chezmoi.keepassxc.command` can point to an arbitrary binary. We derive the test config from `chezmoi dump-config --format json`, patch the keepassxc section with jq, and pass it via `--config / --config-format json`. This is more surgical than prepending to PATH: it doesn't affect any other tools running in the same shell, and the config origin is explicit.

*Alternative considered*: prepend `tests/bin` to `$PATH`. Rejected because it silently affects any other command that runs during the session.

**D2: `prompt = false` to suppress TTY opening**

Setting `keepassxc.prompt = false` in the test config tells chezmoi to skip opening `/dev/tty` for the password. Without this, chezmoi tries to open a TTY before even spawning the binary, failing in non-interactive contexts regardless of what our mock does.

**D3: Auto-generate `mock:<entry>:<attr>` for unknown lookups**

Rather than require every entry to be pre-populated in a fixture file, the mock returns `mock:<entry>:<attr>` for anything not found. This means new `keepassxcAttribute` calls in templates "just work" without any fixture maintenance. The format is obviously mock, so accidental use of test values is easy to spot.

*Alternative considered*: exit non-zero on unknown entry. Rejected because it would force fixture updates on every new template addition.

**D4: Small fixture file for format-sensitive entries only**

Some attributes must conform to a specific format for the template to render correctly (SSH public key lines, PEM private key blocks, `known_hosts` format, JSON arrays). These are pinned in `tests/fixtures/keepassxc.json`. The fixture file is optional — if absent the mock still works, just without pinned values.

**D5: `eval "tokens=($line)"` to parse shell-quoted commands**

Chezmoi sends commands like `show "SSH (Mac Studio)" --attributes id_ed25519.pub`. Entry names contain spaces and are double-quoted. `read -ra` splits on whitespace and breaks these names. `eval "tokens=($line)"` interprets shell quoting correctly. In this context the input is always generated by chezmoi (not user-supplied), so eval is safe.

## Risks / Trade-offs

[eval on chezmoi-generated input] Shell injection if chezmoi ever includes user-controlled data in command tokens (e.g. an entry name containing backticks) → Mitigation: the mock is test-only, never runs against real databases or in production paths; acceptable risk for a dev tool.

[Auto-generated values pass template rendering] Templates that validate secret format (e.g. check for `ssh-ed25519` prefix) will fail on auto-generated values → Mitigation: pin those entries in the fixture file.

[chezmoi protocol may change] The interactive `open` mode command format (`show "<entry>" --attributes <attr> --quiet --show-protected`) could change in a future keepassxc-cli version → Mitigation: the mock is only used via `chezmoi execute-template`; chezmoi controls the command format, and any change would surface immediately as a test failure.

## Migration Plan

No migration needed. The `tests/` directory is additive and isolated from all managed dotfile paths. No existing behavior changes.
