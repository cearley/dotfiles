## ADDED Requirements

### Requirement: brew_env Schema in packages.yaml
The `brew_env` key SHALL be an optional YAML mapping at the category level (e.g., `packages.darwin.work.brew_env`). All keys and values in the mapping SHALL be YAML strings.

#### Scenario: Valid brew_env declaration
- **WHEN** `brew_env` is declared in a category
- **THEN** it SHALL be a YAML mapping (not a list or scalar)
- **AND** every key SHALL be a valid shell variable name (alphanumeric and underscores)
- **AND** every value SHALL be a string

#### Scenario: brew_env coexists with other category keys
- **WHEN** a category declares `brew_env` alongside `brews`, `casks`, or `taps`
- **THEN** the `brews`, `casks`, and `taps` lists SHALL remain plain-string lists
- **AND** no entry in those lists SHALL be changed to a mapping to accommodate env var needs

### Requirement: brew_env Collection in Install Script
The `run_onchange_before_darwin-23-install-packages.sh.tmpl` template SHALL collect `brew_env` maps from all active categories using the same category-iteration logic applied to `brews` and `casks`.

#### Scenario: Template renders export statements for active categories
- **WHEN** the template is rendered
- **AND** one or more active categories contain `brew_env` entries
- **THEN** the rendered script SHALL contain `export KEY="VALUE"` lines for each collected env var
- **AND** these export lines SHALL appear before the `brew bundle` heredoc

#### Scenario: No export statements when brew_env is absent everywhere
- **WHEN** no active category contains a `brew_env` key
- **THEN** the rendered script SHALL contain no `export` statements generated by this feature
- **AND** the `brew bundle` call SHALL be unaffected

#### Scenario: Category iteration order determines last-wins precedence
- **WHEN** the template iterates categories in order (`core`, then `tag_choices` order)
- **AND** two categories define the same `brew_env` key
- **THEN** the value from the later category in iteration order SHALL overwrite the earlier one
- **AND** the final exported value SHALL be from the last active category that defined that key
